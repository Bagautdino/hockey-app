Персоны

Три ключевых типа пользователей платформы с разными целями и путями

Алексей, 38 лет

Родитель / 80% трафика
Отец Димы (10 лет, нападающий, хоккейная школа в Омске). Тратит 50K+ руб./мес. на хоккей. Дима талантлив, но в регионе нет скаутов КХЛ. Алексей тревожится: оправданы ли вложения, конкурентоспособен ли сын на уровне страны.
Цель: Получить объективное подтверждение прогресса ребёнка и сделать его видимым для академий за пределами Омска.

Марина, 45 лет

Скаут / 15% трафика
Селекционер московской хоккейной академии. Ищет перспективных защитников 2014 г.р. из регионов за пределами Москвы и СПб. Сейчас опирается на рекомендации тренеров и редкие выезды на турниры.
Цель: Эффективно находить и оценивать таланты из удалённых регионов без необходимости путешествовать.

Сергей, 52 года

Тренер / Отложен
Главный тренер региональной хоккейной школы. 60 учеников в 3 возрастных группах. Хочет стандартизировать оценки и отслеживать развитие.
Цель: Использовать платформу как внутренний инструмент систематического развития игроков.

Готово

3

Регистрация

Создание аккаунта и выбор роли

Выбор роли и создание аккаунта
Частично
Пользователь выбирает «Родитель» или «Скаут», вводит email и пароль. Форма работает, но выбор роли не влияет на дальнейший опыт.
Осторожный оптимизм
Критический баг: роль не влияет на post-login опыт. Родитель и Скаут видят одинаковый интерфейс. Скаут не получает нужные инструменты (фильтры, избранное).
4

Онбординг (4 шага)

Создание профиля игрока через пошаговый wizard

Шаг 1: Личные данные
Готово
ФИО, дата рождения, позиция, хват, команда, номер, город, регион — 12 полей. Валидация Zod.
Вовлечённость, но начинает чувствовать объём работы
Шаг 2: Антропометрия
Без пропуска
9 параметров: рост, вес, размах рук, ширина плеч, длина ноги, туловище, высота сидя, размер обуви, % жира.
Пик фрикции — большинство родителей не знают размах рук или высоту сидя
Точка отвала №1: Wizard требует 20+ полей без возможности пропуска. Нет кнопки «заполнить позже». Это главный killer конверсии.
Шаг 3: Физические тесты
Без подсказок
11 показателей: бег 20м/60м, прыжки, ловкость, гибкость, отжимания, подтягивания, планка, баланс.
Высокая фрикция — требует уже имеющихся данных тестирования
Нет инструкции как измерять. Родитель без данных тестирования не может пройти дальше.
Шаг 4: Подтверждение и создание
submitForm — no-op
Обзор всех введённых данных, загрузка фото, кнопка «Создать профиль».
Облегчение — но данные на самом деле не сохраняются
submitForm только симулирует задержку и редирект. createPlayer API не вызывается. Все данные теряются.
5

Основной цикл

Повседневное использование платформы

Dashboard
Готово
Рейтинг с дельтой, позиция в регионе, график прогресса, радар навыков, таблица последних тестов.
Момент ценности — если данные осмысленные
Радар навыков и перцентильные бейджи — профессиональная визуализация, которая даёт родителю ощущение объективности.
Показывает только первого игрока. Родители с 2+ детьми не могут переключаться.
Сравнение с другими
Готово
Каталог игроков с поиском, фильтрацией по региону и позиции. Карточки с рейтингом.
Ключевой «aha-момент» — ради этого они пришли
Региональный рейтинг «3-й из 24» — мощный мотиватор для вовлечённых родителей.
Загрузка видео
Заглушка
Backend поддерживает загрузку в MinIO (presigned URL), но UI-плеер — заглушка. Нет формы загрузки.
Инвестиция в платформу — без видео профиль неполный
Добавление новой тест-сессии
Отсутствует
Endpoint POST /api/v1/players/{id}/tests существует, но нет UI для добавления новых результатов тестов.
Без механизма обновления нет повода вернуться
Нет триггера для возврата: ни email, ни уведомлений, ни напоминания «пора провести следующее тестирование».

Путь скаута

8 шагов от обнаружения платформы до контакта с родителем. Скаут никогда не вводит данные — его путь строится вокруг поиска, оценки и действия.

Реализовано
Частично
Заглушка
Отсутствует
1. Осведомлённость
Внешний
Директор академии рассказывает о платформе, или скаут видит упоминание на trackhockey.ru.
Профессиональный интерес
2. Лендинг
Нет блока для скаутов
Лендинг обращается только к родителям. Нет секции «Для скаутов и академий» с отдельным ценностным предложением.
Скептицизм: «Это надёжные данные?»
Скаут не видит свою ценность на лендинге и может уйти.
3. Регистрация как «Скаут»
Идентична родителю
Выбор роли не влияет на дальнейший опыт. Скаут попадает в тот же Dashboard что и родитель.
Недоумение — «зачем мне Dashboard игрока?»
4. Каталог игроков
Нет фильтра по возрасту
Поиск по имени, фильтр по региону и позиции. Но нет фильтра по возрасту — самый критичный фильтр для скаута. Backend поддерживает age_min/age_max, но UI не использует.
Вовлечённость, если фильтры адекватные
Для скаута год рождения — фильтр №1. Без него каталог бесполезен.
5. Профиль игрока
Готово
4 вкладки: био, антропометрия с перцентилями, физтесты, видео. Полноценная карточка игрока.
Оценивание — сравнение с внутренней моделью
Перцентильные бейджи по возрастным нормам — скаут получает объективные ориентиры.
6. Просмотр видео
Заглушка плеера
Видеоплеер отображает заглушку. Без видео платформа теряет главную ценность для скаута — возможность увидеть игрока в действии.
Разочарование — числа без видео не убеждают
Без видео платформа не имеет credibility. Тестовые числа от родителей без визуального подтверждения — ненадёжны.
7. Избранное / Шортлист
Отсутствует
Нет возможности сохранить перспективных игроков. Нет сравнения 2-3 игроков side-by-side.
Фрустрация — «нашёл хорошего игрока, но не могу его запомнить»
8. Связь с родителем
Отсутствует
Нет механизма связи: ни чата, ни отображения контактов, ни экспорта профиля в PDF для передачи директору академии.
Тупик — «нашёл таланта, но ничего не могу сделать»
Вся фаза «действие после обнаружения» отсутствует. Скаут, нашедший игрока, не может предпринять никаких шагов.

Общая диаграмма

Визуализация обоих путей и точки, где они пересекаются в каталоге игроков

Приоритеты изменений

Что нужно исправить, чтобы путь пользователя работал

P0

Починить submitForm — wizard должен сохранять данные

useNewPlayerForm.submitForm() должен вызывать createPlayer API. Сейчас все данные теряются после 4-шагового заполнения.

P0

Сделать антропометрию и тесты необязательными

Родитель должен создать профиль только с базовой информацией (Шаг 1) и заполнять остальное постепенно из профиля игрока.

P0

Разделить опыт Родителя и Скаута после входа

Родитель -> Dashboard. Скаут -> Каталог с расширенными фильтрами. Роль должна определять интерфейс.

P1

Добавить фильтр по возрасту в каталог игроков

Backend уже поддерживает age_min/age_max — нужно подключить в UI. Для скаута это фильтр №1.

P1

Подключить загрузку видео

MinIO presigned URL уже реализован в backend. Нужна форма загрузки и работающий видеоплеер.

P1

UI для добавления новой тест-сессии

POST /api/v1/players/{id}/tests уже работает. Нужна кнопка «Обновить показатели» в Dashboard/профиле.

P2

Шортлист скаута, сравнение игроков, PDF-экспорт

Избранное (localStorage), side-by-side сравнение 2-3 игроков, генерация PDF-профиля для директоров академий.

Гипотезы для валидации

Ключевые допущения, которые необходимо проверить с реальными пользователями

Родители имеют данные тестов

Wizard предполагает, что родитель знает бег 20м своего ребёнка. Реальность: делятся ли хоккейные школы результатами тестирования с родителями?

Рейтинг мотивирует, а не отталкивает

«3-й из 24» воодушевляет. «22-й из 24» может заставить родителя уйти. Нужно ли скрывать ранг от слабых игроков?

Скауты доверяют данным от родителей

Самоотчётные метрики могут быть завышены. Нужен ли механизм верификации — подпись тренера, видео-подтверждение?

Видео — главная фича, не числа

Гипотеза: скауты проигнорируют тестовые баллы, но глубоко вовлекутся в видео. Если да — загрузка видео должна быть приоритетом №1.
